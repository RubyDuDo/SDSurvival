# OfferNotFound 开发计划

## 总体原则

- **原型优先**：先做出能玩的最小版本，验证核心循环是否有趣
- **增量迭代**：每个阶段在上一阶段基础上扩展，不推倒重来
- **技术栈**：Godot 4 + GDScript（快速开发优先，后期性能敏感部分可用 C++ 扩展）

---

## 阶段 0：纸面验证（1-2 天）

**目标**：用纸笔/表格模拟几局，验证"分配能量点"这个循环是否有趣。

- [ ] 用电子表格模拟一局完整游戏（12 回合 = 12 周，先缩短）
- [ ] 确认：能量分配是否有有意义的选择？现金压力是否让人紧张？
- [ ] 记录发现的问题，调整数值

**交付物**：一份数值验证记录，确认核心循环可行或需要调整。

---

## 阶段 1：最小可玩原型（1-2 周）

**目标**：一个能跑起来、能玩完一局的 Godot 项目。纯功能，不管好不好看。

### 范围（做什么）

- 12 回合（12 周，不是 52 周——缩短一局时间方便快速测试）
- 3 个技能：C++、英语、面试技巧
- 每回合 7 点能量，分配给：学习（提升技能）/ 打零工（赚钱）/ 投简历 / 面试
- 3 个岗位：1 个生存岗（外包/兼职）、2 个目标岗（初级开发、中级开发）
- 现金系统：每周固定支出，现金归零 = Game Over
- 1 个结局判定：52 周结束时的状态
- 纯 UI 界面，用 Godot 内置控件

### 范围（不做什么）

- 不做随机事件
- 不做 Buff/Debuff
- 不做多角色选择
- 不做音效和美术
- 不做存档

### 开发任务

- [ ] 项目初始化：Godot 项目结构、场景组织
- [ ] 数据层：GameState（回合数、现金、技能值、能量点）
- [ ] 回合流程：能量分配 → 结算 → 下一回合
- [ ] 岗位系统：岗位数据定义、投递 → 面试 → 录用流程
- [ ] 在职状态：接受 offer 后每周获得收入，但能量被占用
- [ ] UI：主界面（显示状态 + 行动按钮）、回合结算界面、结局界面
- [ ] Game Over / 结局判定

**交付物**：能在 Godot 中运行、玩完一局的原型。

---

## 阶段 2：核心体验完善（1-2 周）

**目标**：让游戏"有感觉"，从能玩变成有点好玩。

- [ ] 回合数扩展到 52（完整一年）
- [ ] 岗位扩展到 8-10 个，分 tier
- [ ] 加入 5-8 个随机事件（裁员、猎头、涨房租等）
- [ ] 加入 3-4 个 Buff/Debuff（焦虑、心流、倦怠等）
- [ ] 技能扩展到 5-6 个
- [ ] 多结局系统（大厂上岸、自由职业、创业、躺平等）
- [ ] 基础 UI 美化：深色主题、IDE 风格

**交付物**：一个内容基本完整、可以给朋友试玩的版本。

---

## 阶段 3：内容与打磨（2-4 周）

**目标**：内容量撑起重玩性，体验打磨到可发布水平。

- [ ] 随机事件扩充到 20-30 个
- [ ] 多角色系统（2-3 个初始角色，不同起始条件）
- [ ] Roguelite 元素：每局解锁新内容/新角色
- [ ] 音效：键盘声、编译声、面试提示音
- [ ] UI 动画与视觉反馈
- [ ] 平衡性调整（基于试玩反馈）
- [ ] 导出 Web(HTML5) 版本

**交付物**：可以发布到 itch.io 的版本。

---

## 阶段 4：发布与迭代（持续）

- [ ] itch.io 发布（Pay What You Want）
- [ ] 收集玩家反馈
- [ ] 根据反馈迭代内容和平衡性
- [ ] 评估是否上 Steam Early Access
