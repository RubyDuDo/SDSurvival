# 求职模拟器 —— 游戏设计分析

## 概念概述

一款以程序员在AI冲击下求职为题材的 **Roguelite 回合制资源管理游戏**。玩家在52周（一年）内通过分配能量点数，管理现金流、提升技能、投递简历、参加面试，最终达成不同的职业结局。

核心体验：**资源分配的两难抉择 + 程序员求职的情绪共鸣**。

---

## 一、可行性分析

### 1.1 为什么这个题材可行

| 维度 | 判断 | 说明 |
|------|------|------|
| 情绪共鸣 | 极强 | 大量程序员正在经历或恐惧AI冲击下的就业困境，切肤之痛 |
| 时效性 | 高 | AI就业焦虑正处高峰期，题材自带话题热度 |
| 自传播性 | 强 | 程序员社区（Reddit/V2EX/Twitter/即刻）会自发讨论和传播 |
| 先例验证 | 有 | *Papers, Please*、*Cart Life* 等证明"苦涩现实题材+小体量"路线可行 |
| 受众消费力 | 强 | 程序员群体集中在Steam，付费意愿高 |

### 1.2 商业可行性

**利好：**
- 开发成本低：纯UI + 简单动画，一个人可以做
- Steam定价 $5-8 合理，破冰门槛低
- 可以先出 Demo / Early Access 验证市场
- 题材自带传播性，营销成本低

**风险：**
- 题材有时效性——需趁AI就业焦虑高峰期尽快推出
- 纯数值游戏容易"算明白就没意思了"——需要足够的随机性和事件内容撑住重玩性
- 受众偏窄（程序员群体），但这个群体消费力强且集中

## 二、玩法方向推荐

### 2.1 品类对比

| 方向 | 代表作 | 适配度 | 理由 |
|------|--------|--------|------|
| **Roguelite 资源管理** | *Cultist Simulator*, *Loot Box Lyfe* | **最高** | 每局30-60分钟，随机事件+策略选择，天然支持重玩 |
| 卡牌构筑 | *Slay the Spire* | 中 | 可把"投简历""面试"做成卡牌，但改造成本高 |
| 经典模拟经营 | *Game Dev Tycoon* | 中低 | 体量偏大，不利于快速试错 |

### 2.2 推荐方向：Roguelite 回合制资源管理

**理由：**

1. **天然支持快速试错** —— 一局30-60分钟，玩家愿意反复尝试不同策略
2. **随机事件是核心乐趣** —— "公司突然裁员""AI替代了你投的岗位""猎头找上门"这些事件本身就是内容
3. **开发量可控** —— 不需要复杂美术，UI驱动即可
4. **每局不同的"build"** —— 专精C++走系统底层路线 vs 全栈走创业路线 vs 英语+沟通走外企路线，形成不同的"流派"

## 三、核心机制设计要点

### 3.1 基本框架

- **一局游戏 = 52回合**（52周 / 一年）
- **每回合决策不超过3-4步**，保证节奏紧凑
- **核心资源**：能量点数（每周固定额度）、现金、技能等级
- **失败条件**：现金归零 = Game Over（现金流是"时钟"机制）
- **胜利衡量**：多元结局，不只是赚钱

### 3.2 技能系统

初版控制在 **6-8个技能**：

- **专业技能（3-4个）**：如 C++、架构能力、前端、数据/AI
- **通用能力（2-3个）**：如 英语、沟通、算法/面试技巧
- **隐藏技能（1个）**：如 人脉（通过特定事件解锁）

### 3.3 岗位系统

初版控制在 **10-15个岗位**，分为两大类：

**技术岗位（分3-4个tier）：**
- 高薪但要求技能匹配度高
- 面试流程长（消耗多回合）
- 是长期目标

**生存岗位：**
- 门槛低，容易拿到
- 给稳定现金流
- **但每周占用3-4个能量点（总共约7个），直接砍掉一半的成长空间**
- 这必须是一个"痛苦但必要的选择"——不能太好也不能太差

### 3.4 求职流程

```
投递简历 → (技能匹配度决定概率) → 收到面试邀请
                                        ↓
                              参加面试（消耗能量）
                                        ↓
                            (面试表现决定概率) → 拿到Offer
                                                    ↓
                                              接受 / 拒绝
```

### 3.5 Buff / Debuff 系统（情绪化设计）

这是让游戏从"纯数值优化"跳出来的关键。

**Buff 示例：**
- **意气风发**：手持3个以上interview时触发，每个能量点可多投1个岗位
- **心流**：连续学习3天，技能提升效率翻倍
- **贵人相助**：通过人脉事件获得内推，跳过简历筛选

**Debuff 示例：**
- **焦虑**：连续被拒3次，能量上限-1
- **倦怠**：在生存岗位连续工作4周以上，学习效率-50%
- **自我怀疑**：面试挂掉高tier岗位，下次面试成功率降低

### 3.6 随机事件

随机事件是重玩性的核心支撑：

- 行业变动类：某领域被AI替代（相关岗位消失）、新赛道兴起（新岗位出现）
- 机遇类：猎头找上门、前同事内推、技术大会免费票
- 危机类：房租上涨、电脑坏了需要维修、突发疾病消耗精力
- 选择类：一个需要 relocate 的高薪offer、自由职业机会

### 3.7 两难抉择（核心乐趣）

设计尽可能多的两难局面：

- 手头只够撑2周，一个大厂终面和一个外包offer同时来，接外包就没精力准备终面
- 技能点是投入"深耕C++"还是"广撒网学前端"
- 一个低薪但学习空间大的岗位 vs 一个高薪但消耗全部精力的岗位
- 花钱报培训班（立即减少现金，但加速技能提升）vs 自学（慢但免费）

### 3.8 多元结局

不要只有"赚多少钱"一个维度，建议设置多种结局：

- 大厂上岸（高薪稳定）
- 远程自由职业（时间自由，收入波动）
- 创业（高风险高回报）
- 成功转行（意外路线）
- 躺平摆烂（也是一种结局——黑色幽默）

## 四、设计红线

- **避免变成Excel模拟器** —— 数值系统要简洁，玩家一眼能算清大概收益，但算不准确切结果（随机性的作用）
- **生存岗位的平衡是关键** —— 必须是痛苦但必要的选择
- **情绪共鸣优先于数值精确** —— buff/debuff 用情绪命名，让玩家在游戏中重温真实求职的心理波动
- **内容量决定寿命** —— 随机事件和岗位种类要足够丰富，否则2-3局就会感到重复

---

## 五、建议开发路线

| 阶段 | 时间 | 内容 |
|------|------|------|
| 纸面原型 | 1-2天 | 用表格模拟几局，验证核心循环是否有趣 |
| 最小可玩版本 | 2-4周 | 3个技能、5个岗位、10个事件、纯文字UI |
| 内测反馈 | 1-2周 | 找程序员朋友试玩，观察他们在哪里做出有趣的决策 |
| 美术包装 + 内容扩充 | 4-6周 | 像素风或简笔画风格，压低美术成本 |
| Steam Early Access | - | 上线收集反馈，持续迭代 |

---

## 六、总结

**Roguelite 回合制资源管理 + 程序员求职题材 = 小体量、强共鸣、可快速验证。**

核心要把"两难抉择"和"情绪化buff/debuff"做好，避免做成纯数值优化题。先用最简版本验证"分配能量点"这个循环本身是否好玩，再堆内容。
